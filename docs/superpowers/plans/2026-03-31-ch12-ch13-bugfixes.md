**Content for each section:**

**Header:** Badge "13", label "Harness / 组合技", title "高阶组合技", intro: "这一章是你的 cookbook — 每个 recipe 都是完整可复制的自动化方案。"

**Failure opening:** "你会用 Hooks，会用子代理，会用 SDK。但'怎么让每个 PR 自动做安全审查' — 你不知道该把哪些组件怎么拼起来。"

**13.1 多维度自动 PR Review:**
- Failure scenario: code review only checks "does it run", security and architecture issues slip through
- Diagnosis: one reviewer can't focus on security + quality + style simultaneously
- Solution architecture (CodeBlock diagram):
  ```
  GitHub PR Event
    → GitHub Actions workflow
      → claude -p (security reviewer) --allowedTools "Read,Grep"
      → claude -p (quality reviewer) --allowedTools "Read,Grep,Bash"
      → claude -p (style reviewer) --allowedTools "Read,Grep"
    → Combine results → Post PR comment → Set status check
  ```
- Complete GitHub Actions workflow yaml (ConfigExample with annotations)
- Three reviewer prompt templates (CodeBlock for each: security, quality, style)
- Result combination script (bash: merge 3 outputs into one structured comment)
- QualityCallout: cost estimate per PR (~$0.15-0.30 for 3 reviewers)
- Verification: run on 10 historical PRs, compare findings vs actual bugs found later

**13.2 大型重构的多代理编排:**
- Failure: 50+ file refactor, single session context collapses at file 30
- Diagnosis: context capacity vs quality tradeoff
- Solution architecture:
  ```
  Plan Mode → Architecture spec (split by module)
    → Module A agent (worktree A) + Hook (test after each file)
    → Module B agent (worktree B) + Hook (test after each file)
    → Module C agent (worktree C) + Hook (test after each file)
    → Main agent: integration verification (cross-module interface check)
    → Merge worktrees
  ```
- Step-by-step walkthrough:
  1. Plan Mode: write the refactoring spec with module boundaries
  2. Agent dispatch: one agent per module with worktree isolation
  3. Per-module Hook: PostToolUse runs module tests after each Write/Edit
  4. Integration: main agent checks interface compatibility
  5. Merge: review each worktree's changes, merge sequentially
- ConfigExample: settings.json with Hook config for per-module testing
- Verification checklist: all tests pass, style consistent, interfaces compatible

**13.3 持续代码健康监控:**
- Failure: tech debt accumulates silently, discovered when it's already severe
- Diagnosis: no one does regular health checks
- Solution:
  - Cron script running `claude -p` daily/weekly
  - Scan prompts: TODO/FIXME count, test coverage delta, dependency age, complexity hotspots
  - Output: structured Markdown report
  - Alert: if any metric crosses threshold → create GitHub Issue automatically
- Complete bash script (CodeBlock): `health-monitor.sh`
- Report template (CodeBlock): Markdown format with sections per metric
- Delivery: `gh issue create` for alerts, file output for reports, optional Channels push (note: research preview)
- Verification: run for 4 weeks, compare tech debt trend before/after

**13.4 设计你自己的组合技:**
- Framework: Problem → Which components → How to wire → How to verify
- Reference back to Ch10 diagnostic tree: start from "what's the problem" → map to Harness components
- Table: common problems → recommended component combinations
- ExerciseCard: identify your team's most painful repetitive workflow, design a combo recipe
