## 三、Ch12 — 进阶：让 Claude 编排 Claude

### 章节定位

| 问题 | 回答 |
|------|------|
| 表达什么 | 从"手动一步步操作"到"让 Claude 自己编排全流程" |
| 整体角色 | Part 5 "进阶"的第一章 — 学完 Harness 基础后的效率飞跃 |
| 观众 | 掌握 Ch01-11 的开发者和想在团队推广结构化 AI 开发流程的 Tech Lead |
| 学到什么 | (1) spec→plan→execute→review→merge 的通用编排模式 (2) superpowers 插件的完整实操 (3) 怎么把这套流程适配到自己团队 |
| 专业且易懂 | 先讲模式（不绑定工具），再用 superpowers 作为推荐实现深度教学，最后教适配 |

### 失败开场

你学了 11 章，会用每个 Harness 组件。但一个中等功能的开发流程你仍然手动编排：自己写 spec、自己开 Plan Mode、一个个派子代理、自己 review 每个产出、自己合并。光"编排"就要 30 分钟。如果 Claude 能把这些步骤也串起来呢？

### 结构

**12.1 从手动到自动：编排模式**

Ch10 教了三个核心原则（Spec 驱动 / 上下文隔离 / 验证循环）。这一节把原则落地为一个**可执行的五阶段流水线**：

```
Spec → Plan → Parallel Execute → Review → Merge
```

- 每个阶段做什么、输入输出是什么
- 每个阶段对应哪个 Ch10 原则（Spec 阶段 = Spec 驱动，Execute 阶段 = 上下文隔离，Review 阶段 = 验证循环）
- 为什么按这个顺序（每个阶段为下个阶段降低不确定性）
- 跳过阶段的条件（小修复可以跳 Spec 直接 Plan → Execute）
- 这个模式不依赖任何特定插件 — 你可以用 Skills + Hooks + Agents 自己搭

**12.2 Superpowers 实战：推荐方案**

立场："这是我们团队验证过的最佳实践，推荐你也用起来。"

对 superpowers 的每个核心 Skill 深度教学：

| Skill | 教什么 |
|-------|--------|
| `/brainstorm` | 怎么引导它问出好问题；怎么在多方案中选择；设计文档的输出格式；什么时候跳过（需求已经很清楚时） |
| `/write-plan` | 计划粒度怎么控制（每步 2-5 分钟）；TDD 步骤怎么写；No Placeholders 原则；计划文档存放位置 |
| subagent-driven-development | 每任务一个新鲜子代理；双阶段 review（spec 合规 → 代码质量）；四种状态处理（DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED）；模型选择策略（机械任务用便宜模型，判断任务用强模型）|
| `/review` | 整体 code review 的触发时机；review 跟 subagent review 的区别（局部 vs 全局）；review 发现问题后的修复循环 |
| finishing-a-development-branch | 三种收尾选择（merge / PR / cleanup）；什么时候直接 merge 什么时候开 PR |

每个 Skill 给：
1. 一个真实交互片段（3-5 轮对话展示）
2. 关键参数/配置说明
3. 常见坑和最佳实践

**12.3 适配到你的团队**

- 哪些阶段必须有、哪些可以省 — 按项目复杂度的决策树
- 怎么用自定义 Skill 替换 superpowers 的某个阶段（比如团队有自己的 review 标准）
- 怎么用 Hook 在阶段之间加门禁（比如 Plan 阶段必须有人 approve 才进入 Execute）
- Plugin 打包分享（折叠 ReferenceSection）：目录结构、元数据、分享方式

**12.4 验证 + 边界**

- 什么时候全自动编排是过度工程（决策树：任务 < 30 分钟 → 手动更快）
- 成本估算：全流程编排一个中等功能约消耗多少 token（分阶段明细）
- ExerciseCard：选一个你最近做过的功能，用 superpowers 重新走一遍，对比耗时和质量

## 四、Ch13 — 高阶组合技

### 章节定位

| 问题 | 回答 |
|------|------|
| 表达什么 | 把 Harness 组件组合成解决特定高价值问题的自动化方案 |
| 整体角色 | Part 5 "进阶"的第二章 — 实战 recipes，拿来就能用 |
| 观众 | 想解决具体自动化问题的开发者和 Tech Lead |
| 学到什么 | 3-4 个端到端的自动化方案，每个都是完整可复制的 |
| 专业且易懂 | 每个组合技遵循 失败→诊断→组合方案→完整代码→验证 结构 |

### 失败开场

你会用 Hooks，会用子代理，会用 SDK。但面对一个真实问题 — 比如"怎么让每个 PR 自动做安全审查" — 你不知道该把哪些组件怎么拼起来。这一章就是你的 cookbook。

### 结构

**13.1 组合技一：多维度自动 PR Review**

- **失败：** 团队的 code review 只看"能不能跑"，安全漏洞和架构问题反复漏过
- **诊断：** 一个 reviewer 不可能同时关注安全、质量、风格、性能。人会疲劳，AI 可以分维度
- **组合方案：**
  - Agent SDK（GitHub Actions 触发）→ 3 个并行子代理（安全 / 质量 / 风格）→ 各自独立 review → 汇总为一条 PR comment → Hook 做最终门禁
  - 每个子代理用不同 prompt 和不同 effort 级别
- **完整代码：** GitHub Actions workflow yaml + 3 个 reviewer Skill 文件 + 汇总脚本
- **验证：** 跑 10 个历史 PR 做回测，对比之前 review 漏过的问题

**13.2 组合技二：大型重构的多代理编排**

- **失败：** 一个涉及 50+ 文件的重构，单 session 做到 30 个文件上下文就崩了
- **诊断：** 上下文容量和质量的矛盾 — 文件越多，每个文件分到的"注意力"越少
- **组合方案：**
  - Plan Mode（先做整体架构设计，输出按模块拆分的子计划）
  - Agent Teams（每个模块一个子代理，worktree 隔离）
  - Hook（每个子代理完成后自动跑测试 + lint）
  - 主代理做集成验证（跨模块接口一致性检查）
- **完整代码：** Plan 模板 + Agent 配置 + Hook 脚本 + 集成验证 prompt
- **验证：** 重构完成后全量测试通过率、代码风格一致性、接口兼容性

**13.3 组合技三：持续代码健康监控**

- **失败：** 技术债在不知不觉中积累，等发现时已经很严重
- **诊断：** 没人定期做代码健康检查，靠人记得太不靠谱
- **组合方案：**
  - Agent SDK + 定时任务（每日/每周自动运行）
  - Claude 扫描：TODO/FIXME 趋势、测试覆盖率变化、依赖过期检查、复杂度热点
  - 输出结构化报告（Markdown 文件），通过 Channels 推送或团队自有通知工具转发（Slack webhook、邮件脚本等 — Claude Code 本身不发邮件）
  - 趋势超阈值自动创建 Issue
- **完整代码：** 定时脚本 + 扫描 prompt + 报告模板 + 告警逻辑
- **验证：** 跑 4 周，对比代码健康指标趋势

**13.4 设计你自己的组合技**

- 组合技设计框架：问题 → 哪些组件能解决 → 怎么串联 → 怎么验证
- 回顾 Ch10 的诊断框架：从"Claude 表现不对"出发，定位需要哪些组件
- ExerciseCard：你团队最头疼的重复性工作是什么？设计一个组合技方案
